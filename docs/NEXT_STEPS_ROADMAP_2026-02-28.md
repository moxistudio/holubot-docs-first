# HoluBot 下一步工作路线图（基于 2026-02-28 评价）

## 1. 路线原则

1. 主仓稳态优先：没有稳定性闭环，不做大规模对外发布。
2. 单周单主目标：每周只追一个核心里程碑，其他仓库冻结新功能。
3. 先证据后叙事：先补实测数据，再做 GitHub 与社媒扩散。
4. 开源分层发布：先“文档+最小示例”，后“核心代码”。

## 2. 未来 4 周执行计划

## Week 1：稳态收口（P0）

目标：完成“可对外展示”的最低稳定门槛。

1. 完成 Telegram webhook 切流与回滚演练（T31 收口）
- 交付物：切流记录、回滚记录、72 小时观测日志。
- 验收：告警可定位，5 分钟内可回滚。

2. 补齐周报助手实战样本（T32 收口）
- 交付物：>=2 条真实成功样本 + >=1 条失败样本。
- 验收：失败路径可追溯到步骤级别。

3. 冻结并行仓新功能
- 范围：`HoluWebUI`、`HoluInputCli`、`HoluAgentMaker`。
- 规则：仅允许 bugfix，不新增能力面。

## Week 2：证据补齐（P0）

目标：把“理论优势”补成“可公开的实测证据”。

1. 建立 token/延迟对比基线
- 交付物：对照实验文档（简单问答、搜索、周报、文档处理四类场景）。
- 指标：平均耗时、token、成功率、失败原因分布。

2. 建立统一验收脚本入口（T33 第一版）
- 交付物：`一键回归脚本 + 标准报告模板`。
- 验收：同一命令可重复产出结果。

3. 统一版本矩阵
- 交付物：主仓与三子仓的 schema/API 版本表。
- 验收：每次变更必须更新矩阵。

4. 吸收 CoPaw 的安装体验（P0）
- 交付物：`pip install holubot && holubot init && holubot app` 的 CLI 体验设计稿（命令契约 + 参数表 + 错误码 + 目录结构）。
- 验收：评审通过后，可进入 Week 3 最小实现；文档中明确非目标（暂不追求完整生产安装）。

5. 成本对比实测报告（P0）
- 交付物：`HoluBot vs 直连 Agent` 的成本对比报告（覆盖 4 类场景，含样本输入与输出摘要）。
- 指标：token、延迟、成功率、失败分布；报告内给出“可公开引用”版本。

## Week 3：开源首发（P0）

目标：低风险完成 GitHub 首次公开。

1. 发布内容范围（第一阶段）
- 白皮书脱敏版
- 架构图
- `agent.yaml` 规范
- 最小可运行示例（如 standup/weekly_report）
- 英文 README + 中文 README

2. 安全与合规清理
- 检查项：`.env`、密钥、域名/IP、个人数据、数据库样本。
- 验收：通过仓库脱敏检查清单。

3. 仓库基础设施
- `LICENSE`、`CONTRIBUTING`、`SECURITY`、Issue/PR 模板、`v0.1.0-alpha` tag。

4. README 工程升级（吸收 CoPaw 优点，P0）
- 交付物：README 首屏重构（痛点 Hook、架构图、Quick Start、双语导航、路线图链接、FAQ）。
- 验收：陌生开发者 30 秒内可回答“这是什么/解决什么/下一步如何关注”。

5. 本地小模型选项设计（P1）
- 交付物：前台本地模型方案说明（llama.cpp/MLX 二选一优先级、资源需求、降级策略、开关配置）。
- 验收：给出 PoC 路径与风险清单，不阻塞 Week 3 首发。

## Week 4：传播与反馈回路（P1）

目标：形成“开源 -> 反馈 -> 迭代”的外部循环。

1. X 首发 thread（英文）
- 结构：问题（Agent 安全治理缺口）-> 架构 -> 代码/文档链接。

2. 小红书首发（中文）
- 结构：个人实战案例（周报、文档填充、三通道）+ 真实截图。

3. 反馈分流
- 交付物：反馈看板（Bug / Feature / 文档 / 商业线索）。
- 验收：每周一次 triage，输出下周迭代列表。

4. 安全审计报告（做 CoPaw 做不了的事，P0）
- 交付物：`CoPaw/OpenClaw Skill 导入安全性分析` 报告（基于 HoluInputCli 三层审核引擎，含样本、规则命中、风险等级）。
- 验收：可公开发布，结论与样本可复现，不包含敏感信息。

5. 治理能力 Demo（P0）
- 交付物：30-45 秒演示视频（危险操作触发审批 -> 用户拒绝 -> 操作阻断 -> 审计留痕）。
- 验收：演示脚本、录屏、字幕三件套齐全，可直接用于 X/GitHub 展示。

## 3. 品牌建设并行任务（轻量，不阻塞主线）

1. 品牌基线（Week 1 完成）
- 固定 slogan：`The Bot with Boundaries`。
- 固定关键词：治理、边界、轻量、可控。
- 固定基础配色与字体规范。

2. Logo/图标（Week 2-3 完成）
- 先做 3 个方向草案（葫芦抽象、边界感、前台/后端双球隐喻）。
- 输出 4 套资源：全彩、单色、方形头像、favicon。

3. GitHub 视觉装修（Week 3 完成）
- Social preview 图
- README 首屏价值表达（5 秒可理解）

## 4. 风险与防漂移机制

1. 多仓并发风险
- 机制：每周一次“主仓合并窗”，非窗口不做跨仓协议变更。

2. 路线漂移风险
- 机制：每周复核“本周是否提升稳定性/证据/可见度”三项指标。

3. 对外节奏过载
- 机制：严格按“GitHub 先行 -> X -> 小红书”，不并发首发。

## 5. 里程碑与关门条件

1. M1（稳定可用）
- 条件：T31/T32 收口 + 72h 稳态通过。

2. M2（可证明）
- 条件：token/延迟实测报告完成 + 一键回归第一版可用。

3. M3（可公开）
- 条件：GitHub 首发完成（脱敏+文档+示例）。

4. M4（可增长）
- 条件：X/小红书首发完成，形成每周反馈迭代节奏。
